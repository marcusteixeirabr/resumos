UA1 — Introdução à Arquitetura de Sistemas

Arquitetura de software é o conjunto de decisões estruturais que definem como um sistema é organizado, quais componentes o compõem e como eles se comunicam.

O que é Arquitetura de Software?

É a estrutura fundamental de um sistema: seus componentes, as relações entre eles e os princípios que guiam seu design e evolução. Escolher a arquitetura certa impacta diretamente manutenibilidade, escalabilidade e desempenho.

Principais Estilos Arquiteturais

🏛️ Camadas (Layered)

Organiza o sistema em camadas (apresentação, negócio, dados). Cada camada só se comunica com a adjacente.

🔌 Microsserviços

Divide o sistema em serviços pequenos, independentes e deployáveis separadamente.

💻 Cliente-Servidor

Divide papéis em cliente (consome) e servidor (fornece). Base da web.

📡 Orientada a Eventos

Componentes se comunicam via eventos assíncronos. Alta desacoplagem.

🧩 SOA

Serviços de negócio reutilizáveis que se comunicam via protocolos padrão (ex.: SOAP, REST).

🔄 MVC

Model-View-Controller separa dados, interface e lógica de controle.

Arquitetura em Camadas — detalhe

┌──────────────────────────┐ │ Camada de Apresentação │ ← Interface com o usuário ├──────────────────────────┤ │ Camada de Negócio │ ← Regras e lógica da aplicação ├──────────────────────────┤ │ Camada de Dados │ ← Acesso e persistência └──────────────────────────┘

Como escolher?

PrioridadeArquitetura recomendada
Organização / manutençãoCamadas
Escalabilidade independenteMicrosserviços
Integração de sistemas legadosSOA
Processamento em tempo realOrientada a Eventos
Aplicações web convencionaisMVC / Cliente-Servidor
Dica de prova

Um sistema pode adotar mais de uma arquitetura simultaneamente. Ex.: uma aplicação pode ser Cliente-Servidor por fora e usar MVC internamente.

Fase de Projeto

UA2 — Fase de Projeto (Modelagem)

A fase de projeto transforma os requisitos levantados em uma solução técnica detalhada antes que qualquer código seja escrito.

Engenharia de Software — Fases

O desenvolvimento de software segue um ciclo de vida. A fase de projeto/modelagem ocorre após a análise de requisitos e antes da implementação:

Requisitos → Análise → [ Projeto / Modelagem ] → Implementação → Testes → Implantação

O que acontece na fase de Projeto?

  • Projeto Arquitetural: escolha do estilo arquitetural e divisão em módulos/subsistemas.
  • Projeto de Interface: como os componentes se comunicam entre si e com o usuário.
  • Projeto de Dados: estrutura do banco de dados (tabelas, entidades, relacionamentos).
  • Projeto Procedimental: detalhamento de algoritmos e lógica interna dos componentes.

Modelos usados na fase de Projeto

ModeloO que representa
Diagrama de ClassesEstrutura estática: classes, atributos, métodos e relacionamentos
Diagrama de SequênciaInteração entre objetos ao longo do tempo
Diagrama de ComponentesOrganização física dos módulos do sistema
DER / MEREstrutura do banco de dados
ProtótiposRepresentação visual das interfaces
Por que modelar antes de codar?

Corrigir um erro de design no papel custa muito menos do que reescrever código em produção. A modelagem reduz retrabalho e facilita a comunicação com a equipe.

Intro à Arquitetura Plano de Projeto

UA3 — Analisar e Desenvolver Plano de Projeto

O Plano de Projeto de Software (PPS) é o documento que organiza como o projeto será executado: o quê, quem, quando e com quais recursos.

Estrutura do Plano de Projeto

  • Escopo: o que está incluído (e o que não está) no projeto.
  • Cronograma: estimativa de tempo por atividade (Gráfico de Gantt, EAP).
  • Recursos: equipe, ferramentas, infraestrutura necessária.
  • Riscos: identificação de ameaças e planos de mitigação.
  • Qualidade: critérios e métricas de aceitação.
  • Comunicação: como e com que frequência a equipe e stakeholders se alinham.

EAP — Estrutura Analítica do Projeto

Decomposição hierárquica do trabalho em pacotes menores e gerenciáveis:

Projeto / \ Módulo A Módulo B / \ \ Tarefa1 Tarefa2 Tarefa3

Estimativas

Pontos de Função

Mede o tamanho funcional do software independente da tecnologia.

Story Points

Usado em ágil; representa esforço relativo entre tarefas.

COCOMO

Modelo matemático de estimativa de custo baseado em linhas de código.

Fase de Projeto Projetar a Solução

UA4 — Projetar a Solução

Com os requisitos mapeados e o plano definido, a fase de projeto da solução detalha tecnicamente como o sistema vai funcionar.

Atividades do Projeto de Solução

  • Definir a arquitetura: escolher o(s) estilo(s) arquiteturais adequados.
  • Projetar os módulos: responsabilidades, interfaces e dependências de cada parte.
  • Projetar o banco de dados: criar o MER/DER.
  • Projetar as interfaces: wireframes e protótipos de baixa/alta fidelidade.
  • Documentar: registrar as decisões em diagramas UML.

Princípios de Bom Design

Alta Coesão

Cada módulo faz apenas uma coisa bem feita.

Baixo Acoplamento

Módulos dependem pouco uns dos outros — facilita mudanças.

DRY

Don't Repeat Yourself — evitar duplicação de lógica.

KISS

Keep It Simple — solução mais simples que funcione.

Padrões de Projeto (Design Patterns)

Soluções reutilizáveis para problemas recorrentes no design de software (GoF — Gang of Four):

CategoriaExemplosObjetivo
CriacionaisSingleton, Factory, BuilderComo criar objetos
EstruturaisAdapter, Decorator, FacadeComo compor estruturas
ComportamentaisObserver, Strategy, CommandComo objetos interagem
Plano de Projeto Arquitetura SOA

UA5 — Arquitetura Orientada a Serviços (SOA)

SOA (Service-Oriented Architecture) organiza o sistema como uma coleção de serviços de negócio reutilizáveis e interoperáveis.

Conceitos Fundamentais

  • Serviço: unidade funcional autônoma com interface bem definida (ex.: "calcularFrete", "autenticarUsuário").
  • Contrato: definição formal da interface do serviço (o que recebe, o que retorna).
  • Interoperabilidade: serviços de diferentes tecnologias conseguem se comunicar.
  • Reutilização: o mesmo serviço pode ser consumido por múltiplas aplicações.

Princípios SOA

Baixo Acoplamento

Serviços independentes; mudanças em um não quebram outro.

Abstração

Consumidor não precisa saber como o serviço funciona internamente.

Reutilização

Lógica de negócio exposta como serviço compartilhado.

Composição

Serviços simples se combinam para criar processos complexos.

SOA vs Microsserviços

AspectoSOAMicrosserviços
GranularidadeServiços de negócio (maiores)Serviços muito pequenos
ComunicaçãoESB (Enterprise Service Bus)APIs REST / mensageria
DeployGeralmente centralizadoIndependente por serviço
Uso típicoIntegração corporativaAplicações cloud-native

UA6 — Projeto de Serviços Web

Serviços Web permitem a comunicação entre sistemas via protocolos da internet (HTTP). Os dois estilos dominantes são SOAP e REST.

SOAP

Simple Object Access Protocol — protocolo formal baseado em XML com contrato rígido (WSDL).

UA7 — Modelo de Análise de Software

O modelo de análise transforma os requisitos em uma representação técnica do sistema, servindo de ponte entre o problema e a solução.

Objetivos da Análise

  • Compreender o domínio do problema.
  • Identificar as entidades, comportamentos e interações do sistema.
  • Criar modelos que sirvam de base para o projeto detalhado.

Tipos de Modelos

Modelo Funcional

O que o sistema faz. Representado por Casos de Uso e Diagramas de Atividades.

Modelo Estrutural

Como o sistema é organizado. Representado por Diagrama de Classes.

Modelo Comportamental

Como o sistema age ao longo do tempo. Representado por Diagrama de Sequência e de Estados.

UML — Unified Modeling Language

Linguagem padrão para modelagem de sistemas orientados a objetos. Principais diagramas:

DiagramaTipoMostra
ClassesEstruturalClasses e relacionamentos estáticos
Casos de UsoComportamentalFuncionalidades e atores
SequênciaComportamentalTroca de mensagens entre objetos
AtividadesComportamentalFluxo de trabalho / algoritmo
ComponentesEstruturalMódulos e dependências físicas
ImplantaçãoEstruturalDistribuição em hardware

UA8 — Modelo de Análise Orientado a Objetos

A análise OO modela o sistema em termos de objetos que possuem dados (atributos) e comportamentos (métodos) e se comunicam entre si.

Pilares da OO

Encapsulamento

Dados e comportamento juntos; detalhes internos ocultos.

Herança

Classes filhas reutilizam e estendem comportamentos da classe pai.

Polimorfismo

Um método se comporta de formas diferentes conforme o objeto.

Abstração

Modelar apenas o que é relevante para o contexto.

Relacionamentos em UML

TipoSímboloSignificado
Associação——Objetos se conhecem
Agregação——◇"Tem um" (parte existe sem o todo)
Composição——◆"Tem um" (parte depende do todo)
Herança——▷"É um" (generalização)
Interface- - ▷"Implementa"
Dependência- - →Usa temporariamente

Identificando Classes

Técnica: ler a especificação e sublinhar os substantivos (candidatos a classes/atributos) e os verbos (candidatos a métodos/associações).

Exemplo rápido

"Um cliente realiza um pedido que contém vários produtos."
→ Classes: Cliente, Pedido, Produto. Relacionamento: Cliente —(1)—→(N)— Pedido —(N)—→(N)— Produto

UA9 — Prototipação e Concepção

Protótipos são representações parciais do sistema criadas para validar ideias com usuários antes de investir em desenvolvimento.

Para que Prototipar?

  • Validar requisitos de interface com stakeholders.
  • Detectar problemas de usabilidade cedo.
  • Alinhar expectativas entre equipe e cliente.
  • Reduzir retrabalho na implementação.

Tipos de Protótipos

TipoFidelidadeFerramentasQuando usar
Esboço (Sketch)Muito baixaPapel e canetaInício, brainstorming
WireframeBaixaFigma, BalsamiqEstrutura e fluxo
MockupMédiaFigma, Adobe XDVisual e identidade
Protótipo navegávelAltaFigma, MarvelTeste de usabilidade

Abordagens

Throwaway

Protótipo descartável — apenas para validação, não vira código.

Evolutivo

Protótipo que é refinado e evolui até virar o produto final.

Prototipação no Design Thinking

No ciclo Empatia → Definição → Ideação → Prototipação → Teste, o protótipo é o artefato que torna a ideia tangível para ser testada com usuários reais.

Análise OO Protótipos no Design

UA10 — Utilizando Protótipos no Design

Protótipos de alta fidelidade permitem que os usuários avaliem o software de forma prática e realista antes da entrega final.

Protótipo de Alta Fidelidade

Representa o produto com aparência e comportamento próximos do real: cores, tipografia, interações e transições funcionando. Ideal para:

  • Testes de usabilidade com usuários finais.
  • Aprovação visual do cliente.
  • Guia de referência para os desenvolvedores.
  • Apresentações e demos.

Processo de Prototipação

Esboços → Wireframes → Mockup → Protótipo Navegável → Testes → Refinamento

Boas Práticas

  • Consistência visual: use sistemas de design (design systems) com componentes reutilizáveis.
  • Mobile first: projete primeiro para telas pequenas, expanda para maiores.
  • Hierarquia visual: o olho do usuário precisa de uma ordem de leitura clara.
  • Feedback visual: estados de hover, loading, erro e sucesso devem estar no protótipo.
  • Acessibilidade: contraste de cores, tamanho de fonte, áreas de toque adequadas.

Ferramentas populares

Figma

Colaborativo, web-based, padrão da indústria. Gratuito para uso básico.

Adobe XD

Integrado ao ecossistema Adobe. Bom para prototipação interativa.

Balsamiq

Foco em wireframes rápidos e de baixa fidelidade.

Astah

Foco em diagramas UML com suporte a prototipação.

UA12 — Diagrama de Componentes

O diagrama de componentes mostra como o sistema é dividido em partes físicas (módulos, bibliotecas, executáveis) e como elas se conectam.

Elementos

  • Componente: unidade modular com interface bem definida (ex.: um módulo, DLL, pacote, microserviço).
  • Interface fornecida: o que o componente oferece (— ○).
  • Interface requerida: o que o componente precisa de outros (— ().
  • Dependência: seta tracejada indicando que um componente usa outro.

Exemplo

┌─────────────────┐ ┌─────────────────┐ │ «component» │ │ «component» │ │ Frontend │───────▶│ Backend API │ │ │ REST │ │ └─────────────────┘ └────────┬────────┘ │ ▼ ┌─────────────────┐ │ «component» │ │ Banco de │ │ Dados │ └─────────────────┘

Por que usar?

O diagrama de componentes é útil para:

  • Planejar a estrutura de implantação do sistema.
  • Identificar dependências críticas entre módulos.
  • Comunicar a arquitetura para a equipe de DevOps/Infraestrutura.
  • Avaliar o impacto de trocar uma tecnologia por outra.
Componente vs Classe

Classes representam estrutura lógica. Componentes representam unidades físicas/deployáveis. Um componente geralmente contém várias classes.

UA13 — Diagrama de Sequência

Mostra a interação entre objetos ao longo do tempo — quem chama quem, em que ordem, com que parâmetros e retornos.

Elementos

  • Linha de vida (lifeline): linha vertical tracejada que representa um objeto ao longo do tempo.
  • Mensagem: seta horizontal indicando chamada de método ou evento.
  • Retorno: seta tracejada com o valor retornado.
  • Barra de ativação: retângulo sobre a linha de vida indicando que o objeto está em execução.
  • Fragmentos: alt (if/else), loop, opt (opcional), par (paralelo).

Exemplo — Login

Usuário :Frontend :AuthService :Database │ │ │ │ │── login() ───▶│ │ │ │ │── autenticar()─▶ │ │ │ │── buscaUser() ─▶ │ │ │◀─ userData ─────│ │ │ │ │ │ ┌──────┴──────┐ │ │ │ │ alt │ │ │ │ │ [válido] │── ok() ─▶ │ │ │ [inválido] │── erro()─▶ │ │ └──────┬──────┘ │ │ │◀── resposta ───│ │ │

Dicas

  • Leia o diagrama de cima para baixo (ordem temporal).
  • Use alt para representar condições (if/else).
  • Use loop para repetições.
  • Mantenha o diagrama simples — nem todo método precisa aparecer.

UA14 — Conceitos de Banco de Dados

Banco de dados relacional organiza dados em tabelas com linhas e colunas. SQL é a linguagem para criar, consultar e manipular esses dados.

Conceitos Fundamentais

Tabela (Relação)

Estrutura bidimensional com linhas (registros) e colunas (atributos).

Chave Primária (PK)

Coluna que identifica unicamente cada registro. Nunca nula.

Chave Estrangeira (FK)

Coluna que referencia a PK de outra tabela. Garante integridade referencial.

Índice

Estrutura que acelera buscas em troca de espaço em disco.

SQL — Comandos Essenciais

SELECT (Consultar)

SELECT nome, telefone
FROM Cliente
WHERE mes_aniversario = 'janeiro'
ORDER BY nome;

INSERT (Inserir)

INSERT INTO Cliente (nome, email, telefone)
VALUES ('Marcus', 'marcus@email.com', '(47) 99999-0000');

UPDATE (Atualizar)

UPDATE Produto
SET preco = 29.90
WHERE id = 15;

JOIN (Junção)

SELECT c.nome, p.data, p.status
FROM Cliente c
INNER JOIN Pedido p ON c.id = p.id_cliente
WHERE p.status = 'pendente';

Tipos de JOIN

TipoRetorna
INNER JOINApenas registros com correspondência em ambas as tabelas
LEFT JOINTodos da tabela esquerda + correspondências da direita (null se não houver)
RIGHT JOINTodos da tabela direita + correspondências da esquerda
FULL JOINTodos de ambas as tabelas
Diag. de Sequência DER

UA15 — Diagrama de Entidade e Relacionamento (DER)

O DER é uma representação gráfica da estrutura do banco de dados, mostrando entidades, atributos e como elas se relacionam.

Elementos do DER

SímboloElementoRepresentação
RetânguloEntidadeTabela / objeto do domínio
ElipseAtributoCaracterística da entidade
Elipse sublinhadaChave primáriaAtributo identificador único
LosangoRelacionamentoAssociação entre entidades
Linha com notaçãoCardinalidade1:1, 1:N, N:N

Cardinalidades

1:1 (Um para Um)

Ex.: Pessoa ←→ CPF. Cada pessoa tem um CPF único.

1:N (Um para Muitos)

Ex.: Cliente ←→ Pedidos. Um cliente faz vários pedidos.

N:N (Muitos para Muitos)

Ex.: Pedido ←→ Produto. Um pedido tem vários produtos; um produto aparece em vários pedidos.

Exemplo — Loja Virtual

[Cliente] ──1:N── <realiza> ──N:1── [Pedido] │ N:N │ [Produto] (via ItemPedido)

Relação N:N → Entidade Associativa

Quando há N:N, criamos uma tabela intermediária para resolver a relação:

-- Tabela associativa para Pedido x Produto
CREATE TABLE ItemPedido (
  id_pedido  INT REFERENCES Pedido(id),
  id_produto INT REFERENCES Produto(id),
  quantidade INT,
  preco_unit DECIMAL(10,2),
  PRIMARY KEY (id_pedido, id_produto)
);
Conceitos de BD MER

UA16 — Modelo de Entidades e Relacionamentos (MER)

O MER é a especificação textual e detalhada do banco de dados — define tabelas, colunas, tipos, restrições e relacionamentos para guiar a implementação.

Do DER ao MER

O DER é a visão conceitual (abstrata). O MER é a visão lógica (implementável). Passos para converter:

  1. Cada entidade vira uma tabela.
  2. Cada atributo vira uma coluna com tipo de dado.
  3. O atributo-chave vira a PRIMARY KEY.
  4. Relacionamentos 1:N geram uma FOREIGN KEY na tabela do lado N.
  5. Relacionamentos N:N geram uma tabela associativa.

Tipos de Dados Comuns

TipoUsoExemplo
INT / INTEGERNúmero inteiroid, quantidade
VARCHAR(n)Texto variável até n charsnome, email
DECIMAL(p,s)Número decimal exatopreco, salario
DATEData (YYYY-MM-DD)data_nascimento
DATETIMEData e horacriado_em
BOOLEANVerdadeiro/Falsoativo, verificado
TEXTTexto longodescricao, observacao

Exemplo completo — Imobiliária

CREATE TABLE Proprietario (
  id_proprietario INT PRIMARY KEY AUTO_INCREMENT,
  nome    VARCHAR(100) NOT NULL,
  cpf     VARCHAR(14)  UNIQUE NOT NULL,
  telefone VARCHAR(18)
);

CREATE TABLE Imovel (
  id_imovel  INT PRIMARY KEY AUTO_INCREMENT,
  endereco   VARCHAR(255) NOT NULL,
  tipo       VARCHAR(50),
  preco      DECIMAL(12,2),
  id_prop    INT NOT NULL REFERENCES Proprietario(id_proprietario)
);

CREATE TABLE Contrato (
  id_contrato  INT PRIMARY KEY AUTO_INCREMENT,
  data_inicio  DATE,
  data_fim     DATE,
  id_imovel    INT REFERENCES Imovel(id_imovel),
  id_cliente   INT REFERENCES Cliente(id_cliente),
  id_corretor  INT REFERENCES Corretor(id_corretor)
);
Checklist de um bom MER

✓ Toda tabela tem PRIMARY KEY  |  ✓ FKs garantem integridade  |  ✓ Campos obrigatórios com NOT NULL  |  ✓ Tipos de dados adequados  |  ✓ Sem redundância desnecessária
