UA1 — Introdução à Arquitetura de Sistemas

Arquitetura de software é a estrutura fundamental de um sistema — as decisões estruturais que definem seus componentes, como eles se comunicam e os princípios que guiam sua evolução. É invisível para o usuário final, mas determina tudo.

A Fundação Invisível

Assim como um engenheiro civil precisa de uma planta baixa antes de erguer um prédio, o software exige um esquema estratégico antes de qualquer linha de código. Arquitetura não é o código em si — é a estrutura que organiza o código, guia sua evolução e garante eficiência a longo prazo.

Decisões arquiteturais impactam diretamente desempenho, custo de manutenção, segurança e capacidade de escalar. Ignorar a arquitetura no início gera o que chamamos de débito técnico — problemas caros de corrigir depois.

A Primeira Lei da Arquitetura

"Tudo na arquitetura de software é um trade-off." — Robert C. Martin. Não existe arquitetura perfeita; existem apenas opções otimizadas para contextos específicos. Escolher microsserviços não é "melhor" que monolito — depende do problema.

Arquitetura Monolítica

Toda a aplicação — interface, regras de negócio e dados — é construída como uma única unidade de implantação. É o ponto de partida natural da maioria dos sistemas.

  • Vantagem: desenvolvimento simples, fácil de testar localmente, baixa barreira de entrada.
  • Trade-off: alto acoplamento. Uma mudança em qualquer parte pode exigir recompilação e redeploy de tudo. Dificulta escalonamento horizontal.
  • Quando usar: times pequenos, sistemas novos, MVPs, domínios ainda indefinidos.

Arquitetura em Camadas (Layered)

Organiza o sistema em níveis horizontais isolados, onde cada camada tem uma responsabilidade específica e só se comunica com a camada imediatamente adjacente.

CamadaResponsabilidadeExemplos
ApresentaçãoInterface com o usuárioHTML, React, API Gateway
NegócioRegras e lógica da aplicaçãoServices, Use Cases, Validações
DadosAcesso e persistênciaRepositories, ORM, Cache, BD
  • Vantagem: separação clara de responsabilidades. É possível trocar a interface sem tocar nas regras de negócio.
  • Trade-off: pode criar acoplamento vertical excessivo ("God Layer") se as camadas não forem bem definidas.

Microsserviços

Decompõe a aplicação em serviços pequenos, independentes, altamente focados em capacidades únicas de negócio. Cada serviço tem seu próprio banco de dados e pode ser deployado separadamente.

  • Vantagem: escalabilidade independente por serviço, resiliência sistêmica (falha de um pilar não derruba o prédio), autonomia de equipes.
  • Trade-off: complexidade operacional alta (rede, monitoramento distribuído, consistência eventual). Custo inicial elevado.
  • Quando usar: sistemas com alta demanda diferenciada por módulo, times grandes e autônomos, cloud-native.

Orientada a Eventos (Event-Driven)

Componentes se comunicam de forma assíncrona emitindo e consumindo eventos através de um broker (ex.: Kafka, RabbitMQ). Nenhum componente precisa saber da existência do outro.

  • Vantagem: desacoplamento extremo, alta eficiência para absorver picos de carga.
  • Trade-off: rastreabilidade e debugging são mais difíceis. Consistência eventual exige cuidado.

Arquitetura Cliente-Servidor

Divide o sistema em dois papéis bem definidos: o cliente, que solicita recursos ou serviços, e o servidor, que os processa e responde. É o modelo fundamental da Web e de boa parte das aplicações corporativas.

  • Vantagem: centralização do controle e dos dados no servidor facilita manutenção, backup e aplicação de regras de segurança.
  • Trade-off: o servidor pode se tornar gargalo (single point of failure). Latência de rede afeta a experiência do usuário.
  • Quando usar: aplicações Web, sistemas de e-commerce, portais corporativos — qualquer cenário com muitos clientes acessando dados centralizados.
Saiba mais — MVC (Model-View-Controller)

MVC é um padrão arquitetural que organiza a aplicação em três camadas colaborativas: Model (dados e regras de negócio), View (apresentação ao usuário) e Controller (coordenação entre Model e View). É frequentemente aplicado dentro de uma arquitetura Cliente-Servidor — o servidor implementa Model + Controller enquanto o cliente renderiza a View. Frameworks como Spring MVC, Django e Ruby on Rails popularizaram esse padrão.

Critérios de Escolha Arquitetural

Escolher uma arquitetura sem critérios claros é apostar no escuro. O PDF da disciplina destaca cinco atributos de qualidade que devem guiar essa decisão:

Desempenho

Tempo de resposta, throughput e uso eficiente de recursos. Arquiteturas com muitas camadas de indireção (ex.: microsserviços com latência de rede) podem prejudicar o desempenho se não forem bem dimensionadas. Event-Driven absorve picos bem, mas adiciona latência de processamento assíncrono.

Escalabilidade

Capacidade de crescer sem perda de desempenho — horizontal (mais instâncias) ou vertical (mais hardware). Microsserviços permitem escalar módulos individualmente; monolitos escalam tudo ou nada.

Segurança

Controle de acesso, proteção de dados e isolamento de falhas. A arquitetura define onde as fronteiras de confiança existem. Em microsserviços, a comunicação entre serviços precisa ser autenticada mesmo internamente.

Manutenibilidade

Facilidade de corrigir, evoluir e entender o sistema. Alta coesão e baixo acoplamento são os princípios-chave. Um monolito bem estruturado pode ser mais manutenível que microsserviços mal projetados.

Custo e Tempo de Desenvolvimento

Microsserviços custam mais para montar a infraestrutura inicial (orquestração, CI/CD por serviço, observabilidade distribuída). Para startups e MVPs, monolito ou camadas entregam mais rápido com menos investimento inicial.

Na Prática: dois cenários reais

Cenário 1 — Startup Fintech

Uma startup de pagamentos escolheu microsserviços desde o dia 1 para "estar pronta para escalar". Com um time de 4 pessoas, o overhead de manter múltiplos serviços, pipelines separados e comunicação via API atrasou o lançamento em meses. A lição: começar com monolito modular e migrar para microsserviços quando a escala realmente exigir é uma estratégia válida e frequentemente mais inteligente.

Cenário 2 — E-commerce na Black Friday

Uma plataforma de e-commerce com arquitetura monolítica não suportou o pico de acessos da Black Friday — todo o sistema ficou fora do ar, incluindo o módulo de pagamentos que processava pouquíssimas requisições. Com microsserviços, apenas o serviço de catálogo teria sofrido degradação, mantendo pagamentos e checkout funcionais. O custo do downtime justificou a migração arquitetural.

A Importância da Arquitetura

"A arquitetura representa as decisões de design significativas que moldam um sistema, onde significativo é medido pelo custo de mudança." — Robert C. Martin (2019). Decisões arquiteturais erradas se tornam dívida técnica que compromete a evolução do produto — e corrigi-las depois custa muito mais do que acertar no início.

Fase de Projeto

UA2 — Fase de Projeto (Modelagem)

A fase de projeto transforma requisitos em uma solução técnica detalhada. É a ponte entre "o que o sistema deve fazer" e "como ele vai fazer". Modelar antes de codificar reduz retrabalho e facilita a comunicação da equipe.

O lugar da fase de projeto no ciclo de vida

Após o levantamento e a análise de requisitos, a fase de projeto descreve como o sistema funcionará internamente — arquitetura, linguagens, banco de dados, interface gráfica e outros elementos — de forma que acompanhe os objetivos definidos na análise. O projeto é desenvolvido de forma iterativa: formalidade e detalhes são acrescidos por meio de revisões constantes (Sommerville, 2011).

Não é possível pular a fase de projeto sem pagar o preço depois. Corrigir um erro no papel custa cents; corrigi-lo em produção custa horas ou dias de engenharia.

As quatro atividades do projeto de software

Sommerville (2011) destaca quatro atividades que podem fazer parte do processo de projeto de sistemas de informação:

Projeto Arquitetural

Identifica a estrutura geral do sistema: componentes principais (subsistemas ou módulos), seus relacionamentos e como são distribuídos.

Projeto de Interface

Define as interfaces entre os componentes. Uma interface precisa permite que um componente seja usado sem que outros saibam como ele é implementado.

Projeto de Componente

Projeta o funcionamento interno de cada componente. Pode ser uma declaração de funcionalidade esperada ou um modelo para gerar código automaticamente.

Projeto de Banco de Dados

Projeta as estruturas de dados do sistema e como devem ser representadas e persistidas no banco de dados.

UML — Linguagem de Modelagem Unificada

A UML (Unified Modeling Language) é o padrão adotado para documentar e comunicar decisões de projeto. Fowler (2004) identifica três formas distintas de aplicar a UML:

UML como rascunho

Diagramas informais, frequentemente desenhados à mão em quadros brancos. Usados para explorar partes difíceis do problema antes de decidir a solução.

UML como planta

Diagramas detalhados usados tanto para engenharia reversa (entender código existente) quanto para geração de código (engenharia avante).

UML como linguagem

Especificação executável completa do sistema em UML, da qual o código é gerado automaticamente. Ainda em desenvolvimento teórico e de ferramentas.

Os 5 diagramas UML mais utilizados

Conforme Erickson e Siau (2007), cinco diagramas da UML concentram a maior parte dos casos de uso na prática:

DiagramaO que representa
AtividadesAtividades envolvidas em um processo ou processamento de dados (fluxo de trabalho).
Casos de UsoInterações entre o sistema e seus atores externos (o "quê" o sistema faz).
SequênciaInterações entre atores e sistema, ou entre componentes, ao longo do tempo (ordem das mensagens).
ClassesClasses de objetos no sistema e as associações entre elas (estrutura estática).
EstadoComo o sistema reage a eventos internos e externos; transições entre estados finitos.

Sequência vs. Comunicação: quando usar cada um

Larman (2007) compara dois diagramas de interação frequentemente confundidos:

CritérioDiagrama de SequênciaDiagrama de Comunicação
Ponto forteMostra com clareza a ordem temporal das mensagens; amplas opções de notaçãoEconomia de espaço — novos objetos podem ser adicionados em duas dimensões
Ponto fracoCresce horizontalmente ao acrescentar objetos; consome espaçoMais difícil visualizar a sequência; menos opções de notação
Melhor paraDocumentação formal, leitura de fluxo de chamadas, engenharia reversaRascunho em quadro branco (modelagem ágil), espaço físico limitado

Perspectivas de modelagem (Larman)

A mesma notação UML pode ser usada com três intenções diferentes:

  • Perspectiva conceitual: os diagramas descrevem coisas do mundo real ou do domínio de negócio — útil na fase de análise, antes de pensar em código.
  • Perspectiva de especificação: os diagramas descrevem abstrações de software com interfaces definidas, sem comprometimento com uma tecnologia específica (Java, C#, etc.).
  • Perspectiva de implementação: os diagramas descrevem implementações concretas em uma tecnologia específica, prontas para guiar o desenvolvimento.
Modelagem não é burocracia

O objetivo não é gerar documentação pela documentação. É criar um entendimento compartilhado. Um diagrama desenhado numa reunião de 30 minutos pode evitar semanas de retrabalho. A escolha de quais diagramas usar fica a critério da necessidade do projeto — pode-se usar todos ou apenas alguns dos que a UML disponibiliza (Sommerville, 2011).

Intro à Arquitetura Plano de Projeto

UA3 — Analisar e Desenvolver Plano de Projeto

O plano de projeto documenta tudo que envolve o desenvolvimento de um software: escopo, equipe, infraestrutura, cronograma, riscos e acompanhamento. Sem ele, o projeto está sujeito a instabilidade, retrabalho e custos imprevistos.

O que é o Plano de Projeto?

O projeto de software começa quando termina a primeira iteração da engenharia de requisitos. Seu objetivo é aplicar princípios, conceitos e práticas que levem ao desenvolvimento de um sistema de alta qualidade (Pressman; Maxim, 2016). O plano de projeto é o documento que norteia toda a equipe durante o ciclo de desenvolvimento — definindo métricas, atividades por membro, estimativas e as mudanças que inevitavelmente ocorrem.

Manifesto do Projeto de Software — Mitch Kapor (1991)

"Projeto é onde você fica com um pé em dois mundos — o da tecnologia e o das pessoas. Um software bem projetado apresenta três qualidades: solidez (sem bugs que impeçam seu funcionamento), comodidade (adequado ao propósito para o qual foi planejado) e deleite (a experiência de uso deve ser prazerosa)." — citado em Pressman e Maxim (2016).

Três características de um bom projeto (McLaughlin)

  • Deve implementar todos os requisitos explícitos do modelo de requisitos e acomodar os implícitos desejados pelos envolvidos.
  • Deve ser um guia legível e compreensível para quem gera código, testa e dá suporte ao software.
  • Deve dar uma visão completa do software, tratando os domínios de dados, funcional e comportamental do ponto de vista da implementação.

Custos de um projeto de software

Sommerville (2011) destaca três parâmetros principais para calcular custos:

  • Custo de esforço — o maior componente; engloba o tempo da equipe
  • Custos de hardware e software — licenças, servidores, ferramentas e manutenção
  • Custos de viagens e treinamentos — deslocamentos e capacitação

EAP — Estrutura Analítica do Projeto (WBS)

A EAP decompõe hierarquicamente o trabalho em pacotes gerenciáveis. O resultado é o documento central do plano: WBS (Work Breakdown Structure) com alocação pessoa-tarefa, análise de riscos, cronograma e orçamento. A regra: divida até que cada tarefa dure entre 1 semana e 8–10 semanas — mais que isso deve ser subdividida.

Projeto / \ Módulo A Módulo B / \ \ Tarefa 1 Tarefa 2 Tarefa 3 (2 dias) (3 dias) (1 dia)

Quatro etapas da programação do projeto

  • 1. Identificar atividades — a partir dos requisitos de software e informações de projeto
  • 2. Identificar dependências — quais atividades precisam terminar antes de outras começar
  • 3. Estimar recursos — tempo, pessoas e infraestrutura por atividade
  • 4. Alocar pessoas e criar gráficos — Gráfico de Gantt: barras baseadas em calendário, mostrando responsáveis, duração prevista e datas de início/fim

Técnicas de estimativa

Pontos de Função

Mede o tamanho funcional do software independente de tecnologia ou linguagem.

Story Points

Esforço relativo entre tarefas em Scrum — não horas, mas complexidade comparativa.

COCOMO

Modelo matemático que estima custo a partir do tamanho do sistema (linhas de código).

Planning Poker

Técnica colaborativa: cada membro estima independentemente; divergências são discutidas.

Estimativas são sempre imprecisas — e ainda assim essenciais

Nenhuma estimativa é exata. O valor está no processo: ele força a equipe a pensar sobre o problema antes de codificar. A melhor abordagem combina planejamento formal e métodos ágeis, ajustando o equilíbrio conforme o tipo de projeto (Sommerville, 2011).

Dica de prova — identifique os elementos do plano

Quando o enunciado descrever como o usuário vai interagir com o sistema ou como o software se comunicará com outros sistemas via APIs, o elemento em foco é o Projeto de Interface. Se falar de equipe e papéis, é Organização do Projeto. Se falar de o que pode dar errado, é Análise de Riscos.

Fase de Projeto Projetar a Solução

UA4 — Projetar a Solução

Com os requisitos mapeados e o plano definido, a fase de projeto da solução detalha tecnicamente como o sistema vai ser construído. É aqui que a arquitetura sai do papel e se torna um guia concreto para os desenvolvedores.

Princípios de Bom Design

Alta Coesão

Cada módulo/classe faz uma coisa só e faz bem. Código coeso é mais fácil de entender, testar e reutilizar.

Baixo Acoplamento

Módulos dependem pouco uns dos outros. Alterar um não quebra os outros. Favorece evolução independente.

DRY

Don't Repeat Yourself. Cada pedaço de conhecimento deve ter uma representação única no sistema.

KISS

Keep It Simple. A solução mais simples que funciona é sempre preferível à elegante que ninguém entende.

YAGNI

You Aren't Gonna Need It. Não implemente o que não é necessário agora. Evite over-engineering.

SOLID

5 princípios de design OO (Single Responsibility, Open/Closed, Liskov, Interface Segregation, Dependency Inversion).

SOLID — Detalhe dos princípios

Padrões de Projeto (Design Patterns — GoF)

Soluções reutilizáveis para problemas recorrentes. Não são código — são receitas de design.

CategoriaExemplosResolve
CriacionaisSingleton, Factory, Builder, PrototypeComo criar objetos de forma flexível

Singleton — o mais famoso (e mal usado)

Garante que uma classe tenha apenas uma instância. Útil para configuração, conexão ao banco, loggers. Mas cuidado: Singleton global vira estado global — dificulta testes e paralelismo.

Plano de Projeto Arquitetura SOA

UA5 — Arquitetura Orientada a Serviços (SOA)

SOA organiza o sistema como uma coleção de serviços de negócio reutilizáveis e interoperáveis. Pense em cada serviço como um departamento de uma empresa: autônomo, com responsabilidade clara, mas integrado ao todo.

Conceitos Fundamentais

  • Serviço: unidade funcional autônoma com interface bem definida. Ex.: calcularFrete, autenticarUsuário, emitirNota.
  • Contrato: definição formal do que o serviço aceita e retorna — independente da tecnologia interna.
  • ESB (Enterprise Service Bus): barramento central que roteia mensagens entre serviços em SOA tradicional.
  • Interoperabilidade: serviços em linguagens diferentes conseguem se comunicar via protocolo comum (SOAP/XML ou REST/JSON).

Princípios SOA

Baixo Acoplamento

Serviços independentes — mudanças internas não afetam quem consome.

Abstração

O consumidor não precisa saber como o serviço funciona por dentro.

Reutilização

A mesma lógica de negócio exposta como serviço pode ser consumida por múltiplas aplicações.

Composição

Serviços simples se combinam para construir processos de negócio complexos.

Autonomia

Cada serviço controla sua própria lógica e não depende de outros para funcionar.

Stateless

Idealmente, serviços não guardam estado entre chamadas — torna o sistema mais previsível.

SOA vs. Microsserviços — qual a diferença real?

Microsserviços são frequentemente descritos como "SOA feito certo". As diferenças são principalmente de escopo e implantação:

AspectoSOAMicrosserviços
Granularidade dos serviçosMaiores, orientados a processo de negócioMuito pequenos, focados em capacidade única
ComunicaçãoVia ESB centralizadoDireto via REST/gRPC ou mensageria
Compartilhamento de BDComum (banco compartilhado)Cada serviço tem seu próprio banco
DeployGeralmente centralizado ou em gruposTotalmente independente por serviço
Adoção típicaIntegração corporativa, ERP, sistemas legadosAplicações cloud-native, startups em escala
Por que SOA ainda importa?

A maioria das grandes empresas tem sistemas legados integrados via SOA (SAP, sistemas bancários, ERPs). Entender SOA é essencial para trabalhar com integração corporativa e modernização de sistemas.

Projetar a Solução Serviços Web

UA6 — Projeto de Serviços Web

Serviços Web permitem que sistemas se comuniquem via internet usando protocolos padronizados. Os dois estilos dominantes são SOAP (mais formal) e REST (mais simples e dominante hoje).

SOAP — Simple Object Access Protocol

Protocolo formal baseado em XML com contrato rígido definido pelo WSDL (Web Services Description Language). Muito usado em sistemas corporativos e financeiros onde o contrato precisa ser exato e verificável.

UA7 — Modelo de Análise de Software

O modelo de análise é a representação técnica do sistema que emerge dos requisitos. Ele serve como linguagem comum entre analistas, desenvolvedores e clientes — transformando linguagem natural em estruturas formais.

Por que modelar?

Imagine construir uma casa sem planta. Modelagem de software segue o mesmo princípio: sem modelos, cada desenvolvedor interpreta os requisitos de forma diferente. O resultado é um sistema incoerente.

  • Clareza: torna explícito o que está implícito nos requisitos.
  • Comunicação: cria um vocabulário compartilhado na equipe.
  • Validação: é mais fácil encontrar inconsistências no modelo do que no código.
  • Documentação viva: o modelo é uma documentação que pode evoluir junto com o sistema.

Tipos de Modelos

Modelo Funcional

O que o sistema faz. Casos de Uso, Diagramas de Atividades, User Stories.

Modelo Estrutural

Como o sistema é organizado. Diagrama de Classes, Componentes, Pacotes.

Modelo Comportamental

Como o sistema age ao longo do tempo. Sequência, Estados, Colaboração.

UML — Unified Modeling Language

Linguagem visual padronizada pela OMG para modelar sistemas orientados a objetos. Não é uma metodologia — é uma notação. Você usa os diagramas que fazem sentido para o seu contexto.

DiagramaTipoPergunta que responde
ClassesEstruturalQuais classes existem e como se relacionam?
Casos de UsoComportamentalQuem usa o sistema e para quê?
SequênciaComportamentalQual a ordem das mensagens em um cenário?
AtividadesComportamentalQual o fluxo de um processo ou algoritmo?
ComponentesEstruturalQuais são os módulos físicos e suas dependências?
EstadosComportamentalPor quais estados um objeto pode passar?
ImplantaçãoEstruturalComo o software é distribuído no hardware?
UML não precisa ser formal para ser útil

Um diagrama rabiscado no papel durante uma reunião que resolve uma dúvida da equipe é UML bem aplicada. O valor está na comunicação, não na perfeição da notação.

UA8 — Modelo de Análise Orientado a Objetos

A análise OO modela o sistema em termos de objetos que combinam dados e comportamentos. É a base do design moderno de software e da UML.

Os quatro pilares da OO

Encapsulamento

Dados e comportamento ficam juntos. Detalhes internos ficam ocultos — só a interface é visível. Protege a integridade do objeto.

Herança

Classes filhas reutilizam e estendem comportamentos da classe pai. Elimina duplicação. Cuidado: herança cria acoplamento forte.

Polimorfismo

Um método se comporta diferente dependendo do objeto. animal.falar() retorna "Au!" para Cachorro e "Miau!" para Gato.

Abstração

Modelar apenas o que é relevante para o contexto. Um Carro para um sistema de vendas tem atributos diferentes de um para um sistema de manutenção.

Técnica para identificar classes: substantivos e verbos

Leia o enunciado do problema. Sublinhe os substantivos — candidatos a classes e atributos. Sublinhe os verbos — candidatos a métodos e relacionamentos.

Exemplo prático

"Um cliente realiza um pedido que contém vários produtos. Cada produto tem nome, preço e estoque. O pedido registra a data e o status."

→ Classes: Cliente, Pedido, Produto
→ Atributos: nome, preço, estoque (Produto); data, status (Pedido)
→ Relacionamentos: Cliente 1—N Pedido; Pedido N—N Produto

Tipos de relacionamento em UML

Agregação——◇"Tem um" (parte sobrevive sem o todo)Departamento ◇ Funcionário
Composição——◆"Tem um" (parte depende do todo)Pedido ◆ ItemPedido

Agregação vs. Composição — a diferença prática

Se você deletar o "todo", a "parte" ainda existe? Sim → Agregação. Não → Composição. Deletar um Pedido apaga os ItemPedido? Sim → Composição. Transferir um Funcionário para outro Departamento faz sentido? Sim → Agregação.

UA9 — Prototipação e Concepção

Um protótipo é uma representação parcial do sistema criada para aprender — validar ideias, descobrir problemas de usabilidade e alinhar expectativas antes de investir em desenvolvimento real.

Por que prototipar?

O custo de corrigir um erro cresce exponencialmente ao longo do projeto. Uma interface mal projetada descoberta em um protótipo de papel custa minutos para corrigir. Descoberta em produção, custa semanas de retrabalho e pode destruir a experiência do usuário.

Slides — Do Esboço à Realidade

O Triângulo da Interação

Um bom design de interface equilibra três forças:

  • Usuário: o centro da experiência. Tudo começa nas suas necessidades reais.
  • Contato Físico: como o usuário manipula a interface (toque, clique, gestos).
  • Contato Conceitual: como o usuário interpreta o retorno do sistema — se o que aconteceu faz sentido para ele.

O Circuito de Prototipação

O design não é linear — é um ciclo de aprendizado contínuo:

1. Levantamento → 2. Elaboração do Protótipo → 3. Validação com o Cliente ↑ | └──────────────── 4. Ajustes e Melhorias ←───────────────┘

Cenários: projetar o contexto antes da tela

Antes de desenhar qualquer tela, projete cenários — narrativas informais sobre as tarefas do usuário. Um cenário responde: quem é o usuário, qual é o contexto dele, qual ação ele quer executar e qual é o objetivo final?

O Espectro de Fidelidade

Fidelidade é o grau de semelhança entre o protótipo e o produto final. Não existe nível "certo" — cada fase do projeto pede um nível diferente.

Baixa Fidelidade — papel e caneta

  • Objetivo: validar conceitos e fluxos básicos.
  • Materiais: papel, post-its, caneta, estoques manuais.
  • Vantagem: descartável, rapidíssimo, sem apego emocional — qualquer um consegue riscar e melhorar.
Análise OO Protótipos no Design

UA10 — Utilizando Protótipos no Design

À medida que as ideias se consolidam, o protótipo evolui em fidelidade. Média fidelidade estrutura a informação; alta fidelidade simula a experiência final.

Média Fidelidade — Wireframes

Wireframes são o esqueleto da interface: sem cores, sem tipografia final, sem imagens reais. O foco é na arquitetura da informação — onde cada elemento fica, qual é a hierarquia, como a navegação funciona.

  • Foco na organização visual e nos relacionamentos entre elementos.
  • Simula funções essenciais sem distrações estéticas.
  • Permite testar o fluxo de navegação antes de qualquer decisão visual.
Slides — Níveis de Fidelidade

Alta Fidelidade — simulação da experiência final

Mockups ou protótipos funcionais com aparência e comportamento próximos do produto real. É aqui que se valida com usuários finais antes de escrever código.

  • Cores e UI: paleta de cores, tipografia, ícones, componentes visuais.
  • Interatividade: simula navegação real, transições e tempos de resposta.
  • Validação final: base para o critério de aceite com o cliente antes da implementação.
Quando usar alta fidelidade?

Quando a estrutura e o fluxo já estão validados (wireframe aprovado) e você precisa testar a aparência e a interatividade com usuários reais. Avançar para alta fidelidade antes de validar a estrutura é desperdício — você investe tempo em visual de algo que ainda pode mudar radicalmente.

UA12 — Diagrama de Componentes

O diagrama de componentes mostra a arquitetura física do sistema — como ele é dividido em módulos deployáveis, bibliotecas e executáveis, e como eles dependem uns dos outros.

Elementos do Diagrama de Componentes

  • Componente: unidade modular com interface bem definida. Pode ser um módulo, biblioteca, executável, microserviço ou pacote. Representado por um retângulo com o estereótipo «component».
  • Interface fornecida (lollipop): o que o componente oferece — representada por um círculo sólido (○—).
  • Interface requerida (socket): o que o componente precisa de outros — representada por um semicírculo (—().
  • Dependência: seta tracejada indicando que um componente usa outro.

Exemplo — Aplicação web em 3 camadas

Backend APINode.jsBanco de Dados, Auth ServiceSQL / HTTP

Quando usar?

  • Planejar e comunicar a arquitetura de implantação para DevOps e infraestrutura.
  • Identificar dependências críticas entre módulos — o que quebra se tal componente falhar.
  • Avaliar o impacto de trocar uma tecnologia por outra (ex.: migrar de MySQL para PostgreSQL).
  • Documentar a arquitetura de sistemas já existentes para facilitar onboarding.
Componente ≠ Classe

Classes representam estrutura lógica no nível do código. Componentes representam unidades físicas e deployáveis. Um componente geralmente agrupa dezenas ou centenas de classes que trabalham juntas para uma responsabilidade maior.

UA13 — Diagrama de Sequência

O diagrama de sequência mostra como os objetos interagem ao longo do tempo em um cenário específico. Responde: quem chama quem, em que ordem, com quais parâmetros — e o que retorna.

Elementos fundamentais

  • Participante (lifeline): objeto ou ator representado por uma caixa no topo e uma linha vertical tracejada abaixo.
  • Mensagem síncrona (→): chamada de método que aguarda retorno. Linha sólida com seta sólida.
  • Mensagem assíncrona (⇢): chamada que não aguarda retorno (fire-and-forget). Linha sólida com seta aberta.
  • Retorno (- - →): valor retornado ao chamador. Linha tracejada.
  • Barra de ativação: retângulo estreito sobre a lifeline indicando que o objeto está em execução.
  • Fragmentos combinados: caixas que agrupam mensagens com condições.

Fragmentos combinados — os mais importantes

FragmentoSignificadoEquivalente no código
altAlternativa (com condições)if / else if / else
optOpcional (só ocorre se condição verdadeira)if sem else
loopRepetiçãofor / while
parParalelo (ocorre simultaneamente)Thread / async
refReferência a outro diagramaChamada de função

Leia de cima para baixo (ordem temporal) e da esquerda para a direita (quem inicia a chamada). Cada seta é uma chamada de método ou retorno. As barras de ativação mostram quem está "no controle" em cada momento.

UA14 — Conceitos de Banco de Dados

Banco de dados relacional organiza dados em tabelas com linhas e colunas, usando SQL para criar, consultar e manipular esses dados. É a tecnologia mais usada para persistência de dados em aplicações corporativas.

Slides — Da Planta ao Código: Banco de Dados

Por que banco relacional?

Imagine 30 vendas por mês controladas em planilha — funciona. Agora imagine 40 milhões de transações diárias. A planilha estoura. O banco relacional resolve escalabilidade, consistência e consultas complexas que planilhas jamais conseguiriam.

Conceitos fundamentais

Tabela (Relação)

Estrutura bidimensional: linhas (registros/tuplas) e colunas (atributos/campos).

Chave Primária (PK)

Coluna que identifica unicamente cada registro. Nunca nula, nunca duplicada. O "CPF" da linha.

Chave Estrangeira (FK)

Coluna que referencia a PK de outra tabela. Cria o "elo" entre tabelas e garante integridade referencial.

Índice

Estrutura auxiliar que acelera buscas. Pense como o índice de um livro: você não precisa ler tudo para achar o que quer.

SQL — DDL e DML

SQL tem duas sub-linguagens principais:

  • DDL (Data Definition Language): define a estrutura. Comandos: CREATE TABLE, ALTER TABLE, DROP TABLE.
  • DML (Data Manipulation Language): manipula os dados. Comandos: SELECT, INSERT, UPDATE, DELETE.

SELECT — anatomia completa

SELECT   nome, telefone          -- O Quê: colunas desejadas
FROM     Cliente                 -- De Onde: tabela fonte
WHERE    mes_aniversario = 'janeiro' -- O Filtro: condição
ORDER BY nome ASC;               -- Ordenação

INSERT, UPDATE, DELETE

-- Inserir
INSERT INTO Cliente (nome, email, telefone)
VALUES ('Marcus', 'marcus@email.com', '(47) 99999-0000');

-- Atualizar
UPDATE Produto SET preco = 29.90 WHERE id = 15;

-- Deletar (sempre com WHERE!)
DELETE FROM Pedido WHERE status = 'cancelado' AND data < '2023-01-01';

JOIN — unindo tabelas

SELECT c.nome, p.data, p.status
FROM   Cliente c
INNER JOIN Pedido p ON c.id = p.id_cliente
WHERE  p.status = 'pendente';
Tipo de JOINRetorna
INNER JOINApenas registros com correspondência em ambas as tabelas
LEFT JOINTodos da esquerda + correspondências da direita (null onde não há)
RIGHT JOINTodos da direita + correspondências da esquerda
FULL JOINTodos de ambas as tabelas
DELETE sem WHERE = catástrofe

Nunca execute DELETE FROM tabela sem uma cláusula WHERE. Você apagará todos os registros da tabela. Em produção, isso pode ser irreversível. Sempre teste um SELECT com o mesmo WHERE antes de executar um DELETE ou UPDATE.

UA15 — Diagrama de Entidade e Relacionamento (DER)

O DER é a representação gráfica e conceitual da estrutura do banco de dados. É a "planta baixa" do banco — independe de tecnologia e serve para alinhar o modelo com os requisitos de negócio antes de implementar.

Slides — Modelagem com DER

Elementos do DER

SímboloElementoRepresenta
RetânguloEntidadeObjeto do mundo real que precisa ser armazenado
ElipseAtributoCaracterística de uma entidade
Elipse sublinhadaAtributo-chave (PK)Identificador único da entidade
Elipse duplaAtributo multivaloradoPode ter múltiplos valores (ex.: telefones)
LosangoRelacionamentoAssociação entre entidades
Retângulo duploEntidade fracaDepende de outra para existir

Entidades Físicas vs. Lógicas

  • Físicas (Tangíveis): existem concretamente no mundo real. Ex.: Cliente, Produto, Funcionário, Fornecedor.
  • Lógicas (Conceituais): nascem da interação estrutural para organizar o negócio. Ex.: Venda, Matrícula, Agendamento, Contrato.

Hierarquia de dependência das entidades

  • Entidades Fortes: existem por si só. Ex.: Produto, Cliente.
  • Entidades Fracas: dependem de uma entidade forte para existir. Ex.: Dependente (de Funcionário) — sem o Funcionário, o Dependente não tem sentido.
  • Entidades Associativas: nascem da relação N:N para viabilizar relacionamentos complexos. Ex.: ItemPedido (entre Pedido e Produto).

Cardinalidades

1:1 (Um para Um)

Cada instância de A se relaciona com no máximo uma de B. Ex.: Pessoa ↔ CPF, Presidente ↔ País.

1:N (Um para Muitos)

Uma instância de A se relaciona com várias de B. Ex.: Cliente → Pedidos, Departamento → Funcionários.

N:N (Muitos para Muitos)

Várias de A se relacionam com várias de B. Exige entidade associativa. Ex.: Pedido ↔ Produto.

Exemplo de DER — Loja Virtual

[Cliente] ─1:N─ <realiza> ─N:1─ [Pedido] ─1:N─ <contém> ─N:N─ [Produto] │ │ [ItemPedido] (entidade associativa) - quantidade - preco_unitario
Conceitos de BD MER

UA16 — Modelo de Entidades e Relacionamentos (MER)

O MER é a especificação lógica e implementável do banco de dados. Enquanto o DER é conceitual (independe de tecnologia), o MER detalha tabelas, colunas, tipos de dados e constraints prontos para ser implementados em SQL.

Do DER ao MER — as regras de conversão

  1. Cada entidade forte vira uma tabela.
  2. Cada atributo vira uma coluna com tipo de dado.
  3. O atributo-chave vira a PRIMARY KEY.
  4. Relacionamentos 1:N geram uma FOREIGN KEY na tabela do lado N.
  5. Relacionamentos N:N geram uma tabela associativa com as FKs de ambas as entidades.
  6. Entidades fracas incorporam a PK da entidade forte como FK.

Tipos de dados mais usados

TipoUsoExemplo de coluna
INT / BIGINTNúmeros inteiros. BIGINT para IDs de sistemas com muitos registros.id, quantidade
VARCHAR(n)Texto variável até n caracteres. Mais eficiente que CHAR para dados variáveis.nome, email, cep
TEXTTexto longo sem limite definido.descricao, observacao
DECIMAL(p,s)Número decimal exato (p dígitos totais, s após vírgula). Nunca use FLOAT para dinheiro.preco, salario
DATEData no formato YYYY-MM-DD.data_nascimento, data_pedido
DATETIMEData e hora. Considere TIMESTAMP com timezone para sistemas internacionais.criado_em, atualizado_em
BOOLEANVerdadeiro/Falso (1/0 em muitos SGBDs).ativo, verificado, pago

Constraints essenciais

ConstraintGarante
PRIMARY KEYUnicidade e não-nulidade do identificador
FOREIGN KEYIntegridade referencial — não deixa criar um registro órfão

Exemplo completo — Sistema Imobiliário

CREATE TABLE Proprietario (
  id_proprietario INT          PRIMARY KEY AUTO_INCREMENT,
  nome            VARCHAR(100) NOT NULL,
  cpf             VARCHAR(14)  UNIQUE NOT NULL,
  telefone        VARCHAR(18),
  email           VARCHAR(100) NOT NULL
);

CREATE TABLE Imovel (
  id_imovel       INT            PRIMARY KEY AUTO_INCREMENT,
  endereco        VARCHAR(255)   NOT NULL,
  tipo            VARCHAR(50)    NOT NULL,
  preco           DECIMAL(12,2)  NOT NULL CHECK (preco > 0),
  disponivel      BOOLEAN        DEFAULT TRUE,
  id_proprietario INT            NOT NULL,

CREATE TABLE Contrato (

  tipo         ENUM('venda', 'aluguel') NOT NULL,
  data_inicio  DATE NOT NULL,
  data_fim     DATE,
  valor        DECIMAL(12,2) NOT NULL,

  id_cliente   INT  NOT NULL REFERENCES Cliente(id_cliente),
  id_corretor  INT  REFERENCES Corretor(id_corretor)
);
Checklist de um MER bem feito

✓ Toda tabela tem PRIMARY KEY  |  ✓ FKs definidas para toda relação  |  ✓ Campos obrigatórios com NOT NULL  |  ✓ Tipos de dados adequados (DECIMAL para dinheiro, nunca FLOAT)  |  ✓ Sem redundância: dados que mudam em um lugar só ficam em um lugar  |  ✓ Nomes consistentes (snake_case, singulares para tabelas)
