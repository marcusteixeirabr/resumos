### A Instrução SELECT

A consulta de dados é uma pesquisa realizada no banco de dados para recuperar os registros que atendem a condições preestabelecidas. O instrumento central para isso é a instrução `SELECT`.

**Sintaxe básica:**

```sql
SELECT coluna1, coluna2, ...
FROM tabela
WHERE condição;
```

- `SELECT` — define quais colunas aparecem no resultado
- `FROM` — indica a(s) tabela(s) de origem dos dados
- `WHERE` — filtra as linhas que atendem a uma ou mais condições

Para listar todas as colunas de uma tabela, usa-se o asterisco como curinga:

```sql
SELECT * FROM Professor;
```

---

### Operadores de Comparação

A cláusula `WHERE` aceita os operadores de comparação padrão do SQL:

| Operador | Significado |
|---|---|
| `=` | Igual a |
| `<` | Menor que |
| `>` | Maior que |
| `<=` | Menor que ou igual a |
| `>=` | Maior que ou igual a |
| `<>` ou `!=` | Diferente de |

**Exemplo — recuperar estudantes com média alta:**

```sql
SELECT NomeAluno, SobrenomeAluno, CidadeAluno, MediaAluno
FROM Aluno
WHERE MediaAluno >= 3.7;
```

---

### Expressões nas Cláusulas SELECT e WHERE

É possível usar expressões aritméticas e funções diretamente nas cláusulas `SELECT` e `WHERE`. A palavra-chave `AS` renomeia colunas calculadas:

```sql
-- Aumentar salário em 10% e filtrar por ano de contratação
SELECT NomeProf, SobrenomeProf, CidadeProf,
       SalarioProf * 1.1 AS SalarioReajustado, DataAdmProf
FROM Professor
WHERE YEAR(DataAdmProf) > 1996;
```

Se a coluna calculada não for renomeada com `AS`, a maioria dos SGBDs atribui nomes sem sentido como `Expr001`.

---

### Comparação Inexata com LIKE

O operador `LIKE` permite buscas por padrão em vez de correspondência exata. Os caracteres curinga variam conforme o SGBD:

| Símbolo | Significado | SGBD |
|---|---|---|
| `%` | Qualquer sequência de caracteres | SQL:2003, Oracle, MySQL, PostgreSQL |
| `*` | Qualquer sequência de caracteres | Access (legado) |
| `_` | Qualquer caractere único | SQL:2003 |
| `?` | Qualquer caractere único | Access (legado) |

```sql
-- Cursos SI de nível avançado (Oracle / padrão SQL)
SELECT * FROM Curso WHERE NumCurso LIKE 'SI4%';

-- Cursos com "banco de dados" em qualquer parte da descrição
SELECT * FROM Curso WHERE DescrCurso LIKE '%BANCO DE DADOS%';
```

A distinção entre maiúsculas e minúsculas depende do SGBD: o Oracle faz a distinção por padrão; o Access não.

---

### Operador BETWEEN-AND

`BETWEEN-AND` é um atalho para testar se um valor numérico ou de data está dentro de um intervalo **fechado** (inclui os dois extremos):

```sql
-- Professores contratados em 1999 ou 2000 (Access)
SELECT NomeProf, SobrenomeProf, DataAdmProf
FROM Professor
WHERE DataAdmProf BETWEEN #1/1/1999# AND #12/31/2000#;

-- Equivalente em MySQL/PostgreSQL
SELECT NomeProf, SobrenomeProf, DataAdmProf
FROM Professor
WHERE DataAdmProf BETWEEN '1999-01-01' AND '2000-12-31';
```

`BETWEEN x AND y` é equivalente a `coluna >= x AND coluna <= y`.

---

### Valores Nulos — IS NULL e IS NOT NULL

Quando não existe valor normal para uma coluna, o banco armazena `NULL`. O `NULL` representa "valor desconhecido" ou "não aplicável" — não pode ser comparado com `=`; o teste correto usa `IS NULL` ou `IS NOT NULL`:

```sql
-- Oferecimentos sem instrutor designado no verão de 2006
SELECT NumOfer, NumCurso
FROM Oferecimento
WHERE CPFProf IS NULL
  AND TrimestreOfer = 'VERAO'
  AND AnoOfer = 2006;
```

---

### Expressões Lógicas AND / OR

Ao combinar `AND` e `OR` em uma mesma cláusula `WHERE`, é essencial usar parênteses para deixar explícito o agrupamento — sem eles, o leitor da query pode interpretar a lógica de forma incorreta:

```sql
-- Oferecimentos do outono de 2005 OU do inverno de 2006
SELECT NumOfer, NumCurso, CPFProf
FROM Oferecimento
WHERE (TrimestreOfer = 'OUTONO' AND AnoOfer = 2005)
   OR (TrimestreOfer = 'INVERNO' AND AnoOfer = 2006);
```

O operador `IN` é uma forma mais limpa de expressar múltiplas comparações `OR` sobre a mesma coluna:

```sql
-- Equivalente ao OR acima para uma única coluna
WHERE TurmaAluno IN ('JR', 'SR')
```

### Junção de Tabelas (JOIN)

Quando a consulta precisa de dados de mais de uma tabela, usa-se a **junção**. Há duas formas:

**Forma implícita (produto cartesiano + WHERE)** — padrão Oracle antigo e Access:

```sql
SELECT NumOfer, Oferecimento.NumCurso, NomeProf, SobrenomeProf
FROM Oferecimento, Professor
WHERE TrimestreOfer = 'OUTONO' AND AnoOfer = 2005
  AND ClassificacaoProf = 'ASSISTENTE'
  AND Professor.CPFProf = Oferecimento.CPFProf;
```

A condição de junção (`Professor.CPFProf = Oferecimento.CPFProf`) é expressa na cláusula `WHERE`. Quando uma coluna aparece em mais de uma tabela, o nome deve ser qualificado com o nome da tabela para evitar ambiguidade.

**Forma explícita com INNER JOIN** — padrão SQL:2003, recomendada:

```sql
SELECT NumOfer, NumCurso, NomeProf, SobrenomeProf
FROM Oferecimento
INNER JOIN Professor ON Professor.CPFProf = Oferecimento.CPFProf
WHERE TrimestreOfer = 'OUTONO' AND AnoOfer = 2005
  AND ClassificacaoProf = 'ASSISTENTE';
```

A condição de junção fica na cláusula `ON`, separada dos filtros de linha que ficam no `WHERE`. Isso torna a query mais legível.

### GROUP BY e Funções Agregadas

A cláusula `GROUP BY` agrupa linhas com o mesmo valor em uma ou mais colunas, permitindo calcular resumos por grupo. Toda coluna no `SELECT` que não seja uma função agregada deve aparecer no `GROUP BY`.

**Funções agregadas padrão:**

| Função | Significado |
|---|---|
| `COUNT(*)` | Número de linhas |
| `COUNT(coluna)` | Valores não nulos na coluna |
| `AVG(coluna)` | Média (exclui nulos) |
| `SUM(coluna)` | Soma (exclui nulos) |
| `MIN(coluna)` | Menor valor |
| `MAX(coluna)` | Maior valor |

```sql
-- Média geral de notas por especialização
SELECT Especializacao, AVG(MediaAluno) AS MediaGeral
FROM Aluno
GROUP BY Especializacao;
```

```sql
-- Quantidade de oferecimentos e cursos distintos por ano
SELECT AnoOfer,
       COUNT(*) AS QtdeOferecimentos,
       COUNT(DISTINCT NumCurso) AS QtdeCursos
FROM Oferecimento
GROUP BY AnoOfer;
```

---

### HAVING — Filtrando Grupos

A cláusula `HAVING` aplica condições sobre **grupos** (após o `GROUP BY`), enquanto `WHERE` filtra **linhas individuais** (antes do agrupamento):

- `WHERE` → filtra antes de agrupar; não pode conter funções agregadas
- `HAVING` → filtra depois de agrupar; deve envolver funções agregadas

```sql
-- Média dos estudantes da divisão superior, por especialização,
-- mostrando só as especializações com média > 3.1
SELECT Especializacao, AVG(MediaAluno) AS MediaGeral
FROM Aluno
WHERE TurmaAluno IN ('JR', 'SR')
GROUP BY Especializacao
HAVING AVG(MediaAluno) > 3.1;
```

---

### Ordem de Processamento Lógico do SELECT

A instrução `SELECT` é processada em uma ordem lógica que determina quando cada objeto fica disponível para as cláusulas seguintes. Aliases definidos no `SELECT` (etapa 8), por exemplo, não podem ser usados em `WHERE` (etapa 4) — mas podem ser usados em `ORDER BY` (etapa 10):

| Etapa | Cláusula |
|---|---|
| 1 | `FROM` |
| 2 | `ON` |
| 3 | `JOIN` |
| 4 | `WHERE` |
| 5 | `GROUP BY` |
| 6 | `WITH CUBE` / `WITH ROLLUP` |
| 7 | `HAVING` |
| 8 | `SELECT` |
| 9 | `DISTINCT` |
| 10 | `ORDER BY` |
| 11 | `TOP` / `LIMIT` |

### Práticas Modernas

- **JOIN explícito sempre** — prefira sempre `INNER JOIN ... ON` em vez do produto cartesiano com condição no `WHERE`. O JOIN explícito torna a intenção clara e é menos propenso a erros (esquecer a condição de junção no `WHERE` gera produto cartesiano silenciosamente).
- **LEFT / RIGHT / FULL OUTER JOIN** — para recuperar linhas sem correspondência na outra tabela. Exemplo: `LEFT JOIN` retorna todos os professores, mesmo os que não têm oferecimento. O livro não cobre esses tipos.
- **CTEs (`WITH`)** — Common Table Expressions permitem nomear subconsultas e reutilizá-las na mesma instrução, tornando queries complexas muito mais legíveis:
  ```sql
  WITH professores_senior AS (
    SELECT CPFProf, NomeProf FROM Professor WHERE ClassificacaoProf = 'CATEDRATICO'
  )
  SELECT * FROM Oferecimento JOIN professores_senior USING (CPFProf);
  ```
- **Window Functions** — calculam agregados sem colapsar linhas, usando `OVER(PARTITION BY ... ORDER BY ...)`. Permitem ranking (`RANK`, `ROW_NUMBER`), totais acumulados (`SUM() OVER`) e comparação entre linhas adjacentes (`LAG`, `LEAD`) — impossíveis de fazer limpo só com `GROUP BY`.
- **`EXPLAIN ANALYZE`** — em PostgreSQL e MySQL, mostra o plano de execução real da query e identifica gargalos (table scans em tabelas grandes, joins sem índice). Indispensável ao otimizar consultas lentas.
- **Índices** — colunas frequentemente usadas em `WHERE`, `JOIN ON` e `ORDER BY` devem ter índices. Um `SELECT` lento quase sempre tem como primeira suspeita a ausência de índice na coluna filtrada.
- **`DISTINCT` com cuidado** — `DISTINCT` resolve duplicatas no resultado, mas é sintoma de um JOIN mal formulado ou de ausência de `GROUP BY`. Antes de usar, investigar por que linhas duplicadas estão aparecendo.
- **Datas em SQL moderno** — use tipos `DATE`, `TIMESTAMP` ou `TIMESTAMPTZ` (PostgreSQL) e funções padrão como `EXTRACT(YEAR FROM coluna)` ou `DATE_TRUNC('year', coluna)`. Evite formatos dependentes de SGBD como `#data#` (Access) ou `TO_DATE` do Oracle em código portável.

<Callout type="tip">
**Dicas para a Prova**

- **BETWEEN-AND com datas** — a sintaxe correta (Access) é `BETWEEN #1/1/1999# AND #31/12/2000#` com a cláusula `FROM Professor` obrigatória (questão 1, opção **A**). A opção D está errada porque não tem `FROM`; E está errada porque usa `=` e `AND` sem `BETWEEN`.
- **GROUP BY** — toda coluna no `SELECT` que não é função agregada deve estar no `GROUP BY`. A query correta é `SELECT Especializacao, AVG(MediaAluno) AS MediaGeral FROM Aluno GROUP BY Especializacao` (questão 2, opção **B**). As opções C e D usam coluna errada no `GROUP BY`; a opção A não tem `FROM`.
- **HAVING com condição correta** — para filtrar faturas com valor total acima de 30.000, a resposta é a opção **C**: `FROM Fatura`, `GROUP BY cod_conveniada`, `HAVING sum(vlr_fatura) > 30000`. A opção A tem o valor errado (50.000); B não tem `FROM`; D agrupa pela coluna errada.
- **IS NULL** — para buscar valores nulos usa-se `IS NULL` (não `= NULL`). A query correta é `FROM pessoa_fisica WHERE cod_estado_civil IS NULL` (questão 4, opção **D**). A opção A usa a tabela errada; C usa `IS NOT NULL` (faz o oposto); E não tem `FROM`.
- **MAX / MIN / AVG** — a sintaxe correta usa `AS` para nomear e termina com `FROM tabela;` sem vírgula antes do `FROM` (questão 5, opção **E**). A opção A inverte MAIOR/MENOR; C tem vírgula extra antes do `FROM`; D não tem `FROM`.
- Regra: **WHERE filtra linhas; HAVING filtra grupos**. HAVING exige GROUP BY.
- A **ordem de processamento** do SELECT: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY. Isso explica por que aliases do SELECT não podem ser usados no WHERE.
</Callout>
