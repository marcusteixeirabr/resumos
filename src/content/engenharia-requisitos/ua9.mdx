### O que é gerenciamento de requisitos

Gerenciamento de requisitos é o conjunto de atividades que **mantém a integridade, a acurácia e a atualidade dos acordos sobre requisitos** ao longo de todo o projeto e da vida do produto. Não é uma fase — é uma disciplina contínua.

Enquanto a elicitação, análise e especificação produzem os requisitos, o gerenciamento os **mantém vivos e consistentes** enquanto o produto existe.

### O ciclo de vida de um requisito

Todo requisito passa por estados ao longo da sua vida:

```
Proposto → Em análise → Aprovado → Em desenvolvimento → Implementado → Verificado → Entregue
                ↓                        ↓                    ↓
            Rejeitado               Cancelado              Alterado (inicia novo ciclo)
```

Acompanhar esses estados permite saber, a qualquer momento: quais requisitos já foram entregues, quais estão sendo desenvolvidos, quais foram cancelados e por quê.

### Os quatro pilares do gerenciamento

**1. Controle de versão de requisitos**

Cada requisito tem um identificador estável e um número de versão. Qualquer alteração na descrição, prioridade ou critérios gera uma nova versão — não substitui a anterior sem registro.

Esquema de versão comum:
- 1.0 — versão inicial aprovada
- 1.1 — ajuste menor de texto ou critério
- 2.0 — mudança substantiva no comportamento

**2. Controle de mudanças**

Nenhuma mudança em requisito aprovado ocorre sem processo formal:

```markdown
## Solicitação de Mudança SM-042

| Campo          | Valor                                        |
|----------------|----------------------------------------------|
| ID             | SM-042                                       |
| Data           | 2024-03-15                                   |
| Solicitante    | Ana Silva (Gerência Comercial)               |
| Requisito      | RF-027 — Pagar taxas com cartão de crédito   |
| Mudança        | Adicionar suporte a PIX                      |
| Justificativa  | 68% dos usuários preferem PIX (pesquisa mar/24) |
| Impacto        | RF-027, RF-028, RNF-S04, UC-015, CT-041      |
| Decisão        | Aprovada — sprint 8                          |
```

**3. Acompanhamento de status**

Atributos que definem o estado atual de cada requisito:

| Atributo | Opções típicas |
|---|---|
| Status | Proposto, Em análise, Aprovado, Cancelado, Implementado, Verificado |
| Prioridade | Essencial, Importante, Desejável |
| Versão | 1.0, 1.1, 2.0 |
| Sprint/Release | Sprint 3, Release 1.2 |
| Responsável | Desenvolvedor ou time designado |
| Data-alvo | Quando deve estar entregue |

**4. Rastreabilidade**

A capacidade de seguir um requisito para frente (downstream) e para trás (upstream):

```
RN-001 (Política de comissão)
  └── RF-010 (Calcular comissão)
        ├── UC-005 (Processar folha)
        │     └── CT-031, CT-032, CT-033
        └── RF-011 (Exibir holerite)
              └── UC-006 (Consultar holerite)
                    └── CT-040
```

Se RN-001 mudar, a rastreabilidade mostra imediatamente quais RFs, casos de uso e testes precisam ser revisados.

### SCM — Software Configuration Management

O SCM é a disciplina mais ampla de gerenciamento de configuração de software — controla todos os **objetos de configuração** (itens de configuração) que compõem o produto:

| Objeto de configuração | Exemplos |
|---|---|
| Código-fonte | Arquivos `.java`, `.ts`, `.py` |
| Documentação | ERS, especificações de casos de uso |
| Requisitos | RNs, RFs, RNFs |
| Planos | Cronograma, plano de teste |
| Scripts | Scripts de deploy, migrations de banco |

Quatro atividades do SCM:
1. **Identificação de objetos** — definir quais artefatos estão sob controle de versão
2. **Controle de versão** — usar ferramenta (Git) para registrar histórico de alterações
3. **Controle de alterações** — processo formal para aprovar mudanças (pull requests, code review)
4. **Auditoria de configuração** — verificar que o que foi implementado corresponde ao que foi aprovado

<Callout type="warning">
**Mudanças sem análise de impacto destroem a rastreabilidade**

O cenário mais comum de falha: um stakeholder solicita uma mudança, o desenvolvedor implementa "rapidinho" sem documentar, e nenhuma das ocorrências afetadas (casos de uso, testes, documentação) é atualizada. Seis meses depois, a rastreabilidade está quebrada e ninguém sabe por que certos requisitos foram implementados de certa forma. Toda mudança, por menor que pareça, deve passar pelo processo de controle de mudanças.
</Callout>

### Práticas Modernas

**Docs as Code** — tratar documentação de requisitos como código: versionada em Git, revisada via pull request, com histórico completo de alterações. Ferramentas como MkDocs, Docusaurus e Astro permitem publicar documentação técnica a partir de Markdown versionado em Git.

**Jira + Confluence com links bidirecionais** — histórias de usuário no Jira linkadas a páginas de especificação no Confluence. Quando a história muda de estado, a doc reflete automaticamente.

**Impact analysis automático** — ferramentas de ALM modernas identificam automaticamente quais itens são afetados por uma mudança em um requisito, com base nos links de rastreabilidade cadastrados.

**Feature flags** — mecanismo para entregar código sem ativar funcionalidade. Permite gerenciar a entrega incremental de requisitos aprovados sem criar branches de longa duração.
